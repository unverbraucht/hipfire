## Headline

A3B MoE on gfx906 is **embarrassingly bandwidth-starved** out of the box. Both prefill and decode at master sit at ~25 tok/s for a 3B-active-per-token model. Theoretical HBM-bound ceiling is ~660 tok/s — we're at **4% of peak**. Issue #207 lists 7 gaps; this session attacked Gaps 1, 3, 7 and discovered a **predicate bug** that was a precondition for any of Gap 1's improvements to fire.

**Per-fix attribution (rough):**
- **Predicate fix (`gate_side_mq4` over-strictness):** +12% across the board. This was the dominant win and the precondition for any of Gap 1 to fire. The previous `gate_side_mq4` requirement was making `routed_side_mq4` paths take the slow per-expert `weight_gemv` (with per-expert internal rotation) instead of `gemv_mq4g256_prerotated` reusing a single shared `x_rot_local`.
- **Gap 3 (wave64 `gemv_hfq4g256_residual_sigmoid_scaled_gpu`):** small contribution — 40 launches/token but each kernel is short; the wave64 lift is most visible in launch-count-dominated decode.
- **Predicate fix (`use_gpu_topk` over-strictness) + Gap 1 single-token dp4a:** additional +3-4% on top of v1. The `use_gpu_topk` predicate was identical-in-shape to `gate_side_mq4` and made the Gap 1 dp4a kernels dead code in v1. v2 unblocks them; the +3-4% is the actual dp4a-over-wave64-scalar lift.

## Hypotheses, validated and pending

### ✅ Validated this session

**H1: `gate_side_mq4` predicate is over-strict post-#171.** Confirmed by reading code: the predicate requires router + shared_expert_gate to be MQ4G256, but issue #171 forces those to Q8_F16 in the quantizer. Every shipping MoE quant has `gate_side_mq4=false`, which silently disables `x_rot_local` and forces the routed-expert per-expert loop down the slow `weight_gemv` (per-expert internal rotation) path. **Fixed:** decoupled `x_rot_local` creation from `gate_side_mq4`; now driven by `routed_side_mq4 || gate_side_mq4`.

**H2: `use_gpu_topk` predicate has the same over-strictness.** Confirmed by code reading: `use_gpu_topk = k==8 && gate_side_mq4 && routed_mq4 && routed_gate_up_mq4`. The `gate_side_mq4` requirement here is unjustified — the GPU softmax operates on `router_logits` (regular `[n_exp]` buffer) regardless of which kernel produced it. Comment claimed "atomically-safe softmax-over-routed-logits" — that's wrong. **Fixed:** dropped `gate_side_mq4` from the predicate; `use_gpu_topk` now requires only `k==8 && routed_mq4 && routed_gate_up_mq4 && x_rot_local.is_some()`.

**H3: Gap 1 dp4a kernels work end-to-end.** Validated by smoke-test on full A3B model: output coherent, decode 28-29 tok/s on r1/r2 daemon prompts. **But:** until H2 was fixed, the kernels were unreachable. Re-bench in flight (v2) will verify the lift attributable specifically to dp4a.

**H4: A3B MoE prefill at 25 tok/s is BW-starved.** Validated by code reading + arithmetic. The bottleneck is `gemv_hfq4g256_moe_gate_up_k8_indexed_batched` + `gemv_hfq4g256_moe_down_residual_scaled_k8_indexed_batched` — both are scalar wave64 GEMVs that do **one (row, krank, token) per block** with NO weight-tile reuse across the ~4 tokens that average-routes to the same expert. At K_TOP=8, 256 experts, pp=128 → 4× weight bandwidth waste vs an MMQ-streaming-with-sort-by-expert approach.

### Pending / unvalidated

**H5: Porting the 2 batched MoE indexed kernels to dp4a will give ~20-30% prefill lift.** Untested. The single-token kernels I wrote (Gap 1 part 1 of 4) don't fire in prefill at all — `prefill_moe_ffn_body_batched` unconditionally calls the `_batched` variants. dp4a per-call ALU win (~2× FLOPs/cycle vs hfma2) should translate to ~25% wall-clock if memory traffic isn't the floor. Risk: if these kernels are HBM-bound already (likely at 4% of peak we're not BW-bound, just badly using BW), dp4a doesn't help. Gap 2 (MMQ-streaming-with-sort-by-expert) is the only fix that addresses the fundamental waste.

**H6: Gap 4 (rotation/fusion wave64) is small but easy.** The `mq_rotate_x` and `fused_silu_mul_mq_rotate` kernels are wave32 with `__launch_bounds__(32, 16)` and ds_swizzle butterflies; porting to wave64 needs butterfly-pattern redesign. Issue rates it 0.5-1% decode + 1-2 days. Worth doing after H5 to clean up.

**H7: Gap 5 (prefetch variant for MoE GEMV) translates the +4.8% dense decode lift.** Untested. The HFQ4 dense prefetch kernel works because expert weight rows are contiguous in VRAM (40 layers × 256 experts × 1024 rows / expert × 136 B/group × K/256 groups = fine-grained predictable access). MoE has the same locality. Half-day effort per kernel × 4 = 2 days.

Kernel topology mirrors single-token dp4a + the existing wave64-batched references: `block=[64,1,1]`, `grid=((M+1)/2, K_TOP, N)`, 2 rows/block via warp_id. Xq scratch layout:
- gate_up: `ensure_q8_1_mmq_x(x, n, k)` → `[K/128 × N]`, indexed `Xq[kblock * N + bid]`
- down: `ensure_q8_1_mmq_x(rot_batch, n*k_top, k)` → `[K/128 × (N*K_TOP)]`, indexed `Xq[kblock * (N*K_TOP) + (bid*K_TOP + krank)]`

### What's blocked: `moe_ffn_all_mq4` is the THIRD over-strict predicate

Bench at pp=32 (3 runs, fresh-process, HIPFIRE_PROFILE=1) showed only the single-token `gemv_hfq4g256_moe_{gate_up,down}_indexed_wave64_dp4a` kernels firing — never the new `_batched` variants. Tracing the dispatch:

- `forward_prefill_batch_with_pbs:3919` gates batched prefill on an `eligible` check that requires `moe_ffn_all_mq4(&l.ffn)` for every MoE layer.
- `moe_ffn_all_mq4` (`qwen35.rs:4129`) requires **router AND shared_expert_gate** to be MQ4G256.
- Per issue #171, every shipping MoE quant has those forced to Q8_F16 (Q4 router destroys routing precision).
- So `eligible = false` → prefill falls back to a per-token loop (`qwen35.rs:3932`) that is byte-identical to decode → batched kernels are dead code.

Same pattern as the `gate_side_mq4` and `use_gpu_topk` predicates relaxed last session: an MQ4-only check on `router + shared_expert_gate` that was correct pre-#171 and now permanently blocks the fast path on shipping models.

### Why this is a bigger surgery than the prior two

The previous predicates only gated whether `x_rot_local` got created / whether GPU top-K could run. The batched body at `qwen35.rs:4154` actually **calls `gpu.gemm_hfq4g256` on `ffn.router.buf` and `ffn.shared_expert_gate.buf`** (lines 4202, 4206). Those calls would crash on Q8_F16 weights — `gemm_hfq4g256` reads HFQ4-G256 binary layout.

So relaxing `moe_ffn_all_mq4` to skip the router + shared_expert_gate requirement also requires the batched body to dispatch on dtype at those two call sites — analogous to how the decode path uses the dtype-aware `weight_gemv` helper when `gate_side_mq4` is false (`qwen35.rs:2167`).

### Gap 5 attempt — measured no-op, reverted

After the batched dp4a kernels turned out to be blocked behind `moe_ffn_all_mq4`, attempted the lowest-surgery alternative: port the dense `gemv_hfq4g256_residual_wave64_prefetch.hip` across-quad weight-prefetch pattern onto the single-token dp4a MoE kernels (which **do** fire on shipping A3B — 14% + 9.5% of prefill wall time per the profile bench).

Wrote `gemv_hfq4g256_moe_{gate_up,down}_indexed_wave64_dp4a_prefetch.hip` with across-group (not across-quad — dp4a uses a single-acc per-group loop, not 4-quad unrolled) weight + Q8_1-activation prefetch. Built clean; smoke test passed with byte-identical output and top-5 logits matching to float-noise.

**Profile confirmed both prefetch variants fired (2560× each at pp=32). 3-run bench-cold result: pp32 prefill 30.9 → 31.0 (+0.3%), decode 29.8 → 29.8 (0.0%). Spread 0.3-0.6%. No signal.**

**Why:** the dense prefetch wins because its 4-quad DOG body has 4×8=32 scalar FMAs to hide loads behind. The dp4a kernel's inner body is two `__builtin_amdgcn_sdot4` + 3 FMAs per group — a single packed instruction in place of the FMA chain. Not enough compute latency for prefetched loads to land in L2 before the next iteration's reads. Per-call timing was flat (gate_up 104.0 → 103.6ms; down 70.5 → 71.7ms).

Reverted: kernels + dispatch.rs branching removed. Memory saved as `feedback_dp4a_prefetch_no_op_2026_05_18.md` so future sessions don't re-tread.

Lesson for the Δ ≥ 5% rule: a 0.3% delta on a 0.6%-spread bench is correctly below the action threshold. Per CLAUDE.md, shipping cargo-cult complexity for a no-op is worse than shipping nothing.
