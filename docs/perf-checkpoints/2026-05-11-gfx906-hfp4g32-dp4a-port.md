# gfx906 HFP4G32 dp4a kernel port — perf checkpoint

Date: 2026-05-11
Hardware: AMD Instinct MI50 (gfx906, Vega 20, 60 CUs, 1024 GB/s HBM2 peak)
Model: Qwen 3.5 9B {HFP4G32, MFP4G32, MQ4G256 baseline}
Bench harness: `scripts/bench-cold.sh` (5 fresh-process runs per (model, pp);
asym3 KV; `HIPFIRE_GRAPH=1`; `HIPFIRE_DPM_WARMUP_SECS=10`)

This document closes Phase B' of `/tmp/gfx906_hfp4.md`: the HFP4G32 / MFP4G32
RDNA-optimal FP4 family is now first-class on gfx906 (CDNA1 / Vega 20),
matching the HFQ4G256 production stack within the projected 5–10% perf
band on a 5-run cold-process protocol.

## What changed

Two commits on master:

### `7cbfb46` — HFP4G32 dp4a kernel family

Ported the HFQ4 gfx906 MMQ family + the three fused decode-path wave64
dp4a kernels to HFP4G32, plus a new single-output GEMV.

**Kernels added** (13 files):
- `gemm_hfp4g32_residual_mmq_gfx906_body.cuh` (shared body)
- `gemm_hfp4g32_residual_mmq_gfx906_x{8,16,24,32,40,48,56,64}.hip`
  (8 mmq_x variants; each exports `_x{N}`, `_full_add_x{N}`, `_full_set_x{N}`)
- `fused_qkv_hfp4g32_wave64_dp4a.hip`
- `fused_gate_up_hfp4g32_wave64_dp4a.hip`
- `fused_qkvza_hfp4g32_wave64_dp4a.hip`
- `gemv_hfp4g32_wave64_dp4a.hip` (single-output, for lm_head)

**Three kernel-side diffs from HFQ4 → HFP4** (per the brief in
`/tmp/gfx906_hfp4.md`):
1. Replace `(n - 8)` symmetric INT4 unpack with `kvalues_hfp4 = 2 × E2M1`
   LUT-decoded INT8. ×2 folded into per-block scale at load (×0.5f).
   Matches llama.cpp's MXFP4 pattern in `ggml/src/ggml-cuda/mmq.cuh:867`.
2. Per-block UE8M0 byte replaces FP32 per-256 scale, dequantized via
   FP32 bit-construction (`2^(e-127)` = float with biased exponent = e).
3. FP16 row scale hoisted at scale-collection (folded into the per-block
   scalar that the dp4a inner loop multiplies by).

No zp/sum_x correction (HFP4 is signed-symmetric; the bias term the
HFQ4 kernel carries for its `(n-8)` shift is unnecessary).

**Dispatch wiring**: `gemm_hfp4g32_residual` routes to the new MMQ on
gfx906 at batch ≥ 8. `gemm_qkv_hfp4g32` / `gemm_gate_up_hfp4g32` /
`gemm_qkvza_hfp4g32` route to the fused wave64 dp4a kernels at batch=1.
`gemv_hfp4g32` routes to `gemv_hfp4g32_wave64_dp4a` on gfx906 before the
wave32 fallback.

### `2a374c6` — fused-projection prefill wiring (the 9.4× lift)

After `7cbfb46`, HFP4 pp128 was still 57.5 tok/s — barely faster than
decode. Root cause: `is_batchable_la()` in `qwen35.rs` rejected HFP4G32
on gfx906, sending the entire prefill through the per-token forward_scratch
fallback. Two-line fix to add a `fp4_with_dp4a_gfx906` clause to the
`is_batchable_la` predicate, plus the chained-MMQ dispatch wiring at
the GPU level: `gemm_qkv_hfp4g32` / etc. now call the new
`gemm_hfp4g32_mmq_set_gfx906` 2-4 times per call (sharing one
`ensure_q8_1_mmq_x` prequant scratch), mirroring the production HFQ4
pattern at `dispatch.rs:5249`.

## Diagnostic finding: LUT must live in LDS

A 6-hour rabbit hole during bringup: the new `gemv_hfp4g32_wave64_dp4a`
ran **slower** than the wave32 F32 scalar fallback (0.87× speedup) on the
lm_head shape, despite occupying 32 VGPRs (vs the fallback's 62 + scarce
4-waves/EU occupancy) and emitting 2 `v_dot4_i32_i8` per inner-loop iter
(vs the fallback's 49 v_fma).

Root cause (visible only in compiled ISA, not in ELF metadata): the
`kvalues_hfp4 = 2×E2M1` LUT was declared `__device__ static const
int8_t[16]` (placed in `.rodata`). The gfx906 compiler emitted a
`global_load_ubyte` per nibble — 8 globals per HFP4 block in the inner
loop, dominating the runtime. Trying `__constant__` produced literally
identical ISA: when the LUT index is per-lane (not wave-uniform), the
compiler can't promote to scalar K$ and falls back to global vector
reads.

**Fix**: put the LUT in `__shared__` LDS (16 B), initialize once per WG
by lanes 0..15. The compiler then emits `ds_read_u8` per lookup,
~30× cheaper than a global fetch.

Applied across all four dp4a kernels and the MMQ body. ISA verification
at mmq_x=32:
- `global_load_ubyte` count: 195 → 6 (only weight UE8M0 bytes remain)
- `ds_read_u8` count: 0 → 192 (LUT lookups in LDS)
- VGPR: 88 → 89 (negligible; one extra register for the LDS pointer)
- Spills: 0 → 0

The same LUT pattern lived in `gemv_hfp4g32.hip` (the FP32 fallback)
where the table was correctly placed in `__shared__` from the start.
This is now a load-bearing implementation note for any future dp4a
kernel touching the 2×E2M1 LUT on AMD GCN/CDNA.

## Methodology notes

### Why per-(model, pp) cold-process protocol

Per CLAUDE.md memory `feedback_perf_noise_decode.md` and
`scripts/bench-cold.sh` rationale: within-session A/B on RDNA is ±10-15%
noise from DPM clock state alone, plus an extra 50%-throttle band on
rapid rocprof cycles. 5 fresh-process runs per point with median
reporting plus `HIPFIRE_DPM_WARMUP_SECS=10` collapses the noise band
below 1.5% on every metric collected here.

### Tolerance band per the speed gate

`/tmp/gfx906_hfp4.md` Phase B' Step 6:
> Dense within 10% of HFQ4G256-baseline pp512

Result at pp512: 630.3 / 699.4 = **90.1%** of HFQ4. **Within band.** ✓

The 5–8% prefill regression projected in the brief turns into 9.9% at
pp512 / 9.6% at pp128. Reasons the regression came in at the high end
of the projected band:

1. **HFP4G32's per-row 16-B header.** At K=4096 in 9B Qwen, that's
   16 B / (16 + 4096/32 × 17 B) = 0.6% bandwidth tax per weight row.
   Predictable from the format.
2. **HFP4 has 8× as many scale broadcasts per 256-K group as HFQ4**
   (1 UE8M0 byte every 32 K-elt vs 1 FP32 pair every 256 K-elt). Each
   broadcast costs one LDS slot per row resident-per-window — the MMQ
   body's `x_ds: float × MMQ_Y × 4` (vs HFQ4's `x_dm: float2 × MMQ_Y`)
   adds 1 KiB LDS, marginally lowering occupancy headroom at mmq_x=64.
3. **LUT-decode adds ~2 cycles per byte over `(n - 8)`** in absolute
   terms, but the dp4a inner loop is the same instruction sequence,
   so this term is small.
